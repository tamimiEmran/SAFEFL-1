Windowed BP Redesign: From Memory Crisis to the Next Bottleneck

Why v1 ran out of memory, what the windowed + unary-prior redesign does about it, and the new wall-clock problem it introduced.

1. Original Design — Full-History Factor Graph
One persistent factor graph for the entire training run
v1
Workers are variable nodes — binary latents P(workeri = malicious). Each round, every secure-aggregation group produces one observation, added to the graph as a k-ary factor over its members. The graph is never reset; BP runs over the whole accumulated structure.

Pros: every observation enters the posterior at full weight, no horizon parameter to tune, structure is monotone so JIT compiles once. Cons: graph size scales with round count. The model has no notion of a memory budget.
Design v1 Factor graph Byzantine-robust FL Persistent state
2. What Broke — Linear Growth Hits a Hardware Ceiling
Factor count grows with round count, GPU memory does not
v1 problem
With G groups per round and R rounds, the graph carries G × R k-ary factors. Each holds a 2k-entry message table, with BP allocating factor-to-variable and variable-to-factor messages on every edge. Around round 1300, bp.run raises RESOURCE_EXHAUSTED on a 4 MiB allocation.

Algorithm and implementation are fine. The model just isn't aware that memory is finite.

Root cause: graph size grows linearly with rounds, eventually exceeding the device memory budget.

3. The Redesign — Windowed BP + Unary Priors
Sliding window of K rounds; older evidence collapsed into a per-worker prior
v2
Each round: slide a window over the last K rounds of group observations, snapshot the marginals from round t−K as a per-worker unary prior, build a fresh graph (one unary factor per worker + the windowed k-ary factors), run BP.

The unary prior carries the posterior from round t−K — under a Markov assumption on the latent state, that posterior is a sufficient summary of everything outside the window. Same idea as fixed-lag smoothing. Properties:
  • Memory is O(N + K·G), independent of training length.
  • K trades short-horizon reactivity against historical depth. v1 is K→∞.
  • Older evidence is summarised, not discarded; posterior at round t still reflects rounds 1…t.
Result: no more OOM. Memory footprint is flat across the run.
Design v2 Memory-bounded Factor graph Fixed-lag smoothing
4. New Bottleneck — Per-Round JIT Recompile
Memory crisis solved, wall-clock crisis introduced
v2 problem
A run now takes ~24 h instead of v1's ~5 h. Per-round wall-clock went from ~6 s to ~29 s.

Group assignments rotate every round (security requirement — stable group-mates would let Byzantines collude). Each rotation changes the factor-graph connectivity, so JAX/pgmax JIT-recompiles bp.run from scratch. Compile cost dominates.

v1 — GPU

Graph grows, but structure is monotone — JIT compiles once and amortises. ~6 s/round, mostly BP.

v2 — CPU

New topology every round → fresh JIT compile every round. ~29 s/round, mostly compile.

Why CPU: on GPU the per-round recompile fragmented XLA's BFC allocator and reproduced the OOM at ~round 1300 even with the windowed graph. CPU has the headroom; the recompile cost is the same, just without the allocator dying.
Open: bottleneck is the JIT-compile tax, not memory and not BP.

5. v3 — Detect Once, Then FedAvg
Run v1 for 500 rounds, freeze the partition, switch to FedAvg over the survivors
v3
Use v1 (full-history graph) for the first 500 rounds — well below the OOM threshold and v1's per-round cost is fine. At round 500, threshold the posterior, drop workers flagged as malicious, and aggregate the rest with plain FedAvg for the remainder of training.

Properties:
  • Detection phase pays v1's price (one JIT compile, ~6 s/round, ~25k factors at round 500 — well under the ceiling).
  • Post-500: no factor graph, no BP, no compile. FedAvg is essentially free.
  • Both memory and wall-clock are bounded by construction.
Caveats:
  • One-shot exclusion. A worker that turns Byzantine after round 500 won't be caught.
  • Threshold becomes a new knob; mis-calibration either drops honest workers or keeps malicious ones.
  • Assumes the v1 posterior has stabilised by round 500.
