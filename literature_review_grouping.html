Literature Review: Grouping & Byzantine-Robust Aggregation at Scale

Scope: Byzantine-robust FL aggregation methods, their behaviour with many workers, and the role of grouping/bucketing

1. Bucketing / Grouping as a Pre-Processing Step
Byzantine-Robust Learning on Heterogeneous Datasets via Bucketing
ICLR 2022 Spotlight
Karimireddy, He, Jaggi (EPFL)
Most relevant prior work. Proposes bucketing as a pre-processing step before robust aggregation. Workers are randomly assigned to buckets, gradients are averaged within each bucket, then a robust aggregator (Krum, Median, RFA) is applied to bucket averages.

Motivation: non-IID data heterogeneity causes robust aggregators to fail (honest workers have divergent gradients, aggregator can't distinguish them from Byzantine). Bucketing mixes data distributions across buckets, reducing inter-worker variance ζ².

What they prove: Krum + bucketing is a provably robust aggregator (their Definition A) under heterogeneous data.

What they do NOT study: the IID case where Krum still fails due to minibatch noise at large n. Their experiments use ≤25 workers. The noise reduction from within-bucket averaging (σ² reduction) is mentioned but not the focus — their focus is ζ² (heterogeneity) reduction.

Scope: Byzantine robustness + data heterogeneity. Not framed as a scalability result.

Key distinction: bucketing is introduced as a voluntary algorithmic improvement — the server chooses to bucket for better convergence. No privacy or secure aggregation motivation. The server still sees all individual gradients before bucketing.
Byzantine robustness Grouping/Bucketing Non-IID Theory Voluntary bucketing
Byzantine-Robust Learning on Heterogeneous Data via Gradient Splitting
ICML 2023
Liu, Guo, Cong
Alternative to bucketing: splits gradients into shared and individual components. Addresses the same non-IID failure mode as Karimireddy. Does not use grouping. Scope: Byzantine robustness + non-IID.
Byzantine robustness Non-IID
2. Variance Reduction in Byzantine-Robust FL
Federated Variance-Reduced Stochastic Gradient Descent with Robustness to Byzantine Attacks
arXiv 2019
Wu, Ling, He, He
Recognises that "sizeable SGD-induced stochastic gradient noise makes it challenging to distinguish malicious messages from noisy stochastic gradients." Proposes SAGA-based variance reduction within each worker before aggregation. Scope: Byzantine robustness. Does not use grouping; reduces noise at the source instead.
Byzantine robustness Theory Variance reduction
Byzantine-Robust Federated Learning with Optimal Statistical Rates
AISTATS 2023
Zhu, Wang, Jia
Achieves order-optimal error rates O(σ²/(nm) + ζ²) by combining bucketing with robust mean estimation. Explicitly decomposes error into within-worker noise (σ²) and between-worker heterogeneity (ζ²). Uses bucketing but analyses it through a statistical lens, not a scalability lens.
Byzantine robustness Grouping/Bucketing Theory
3. Benchmarks and Empirical Studies
An Experimental Study of Byzantine-Robust Aggregation Schemes in Federated Learning
IEEE TNNLS 2023
Mhamdi et al.
Comprehensive benchmark of Krum, Multi-Krum, Bulyan, Median, Trim-Mean under FedSGD and FedAvg with various attacks. Concludes "robustness of all aggregation schemes is limited." Does not vary the number of workers as an experimental axis. Does not study grouping. Scope: Byzantine robustness evaluation.
Byzantine robustness Benchmark
Do We Really Need to Design New Byzantine-robust Aggregation Rules?
NDSS 2025
Fang, Nabavirazavi, Cao, Gong
Argues existing rules (Trim-Mean, Median) are sufficient if augmented with server-generated synthetic updates. Does not use grouping/bucketing. Key claim: the problem isn't the aggregation rule, it's the input quality. Scope: security + Byzantine robustness.
Byzantine robustness Security focus

5. Classification of Methods by Type
Selection-based (pick k of n)
Krum — pick 1 closest to majority
Multi-Krum — pick m, average
Bulyan — Krum selection + Trim-Mean

Scaling with n: noise in each candidate grows with smaller per-worker data. Selection quality degrades because all candidates are noisy. More workers ≠ better signal.
Averaging-based (robust statistic over all n)
Coordinate-wise Median
Trimmed Mean
Geometric Median (RFA)
FedAvg (no robustness)

Scaling with n: error ∝ σ²/n. More workers = less noise. These methods inherently benefit from scale.
Note: all existing literature assumes the server sees individual gradients and chooses whether to bucket. Under secure aggregation, the server only sees group outputs — individual gradients are hidden by the protocol. This means (1) robust aggregation must operate on grouped inputs whether or not it helps, (2) the server cannot fall back to ungrouped mode, and (3) the grouping structure is a privacy constraint, not an optimisation choice. No paper in this review studies Byzantine robustness under this constraint.
6. Possible Contribution Angle
Core question: how does the selection vs averaging distinction in robust aggregation interact with worker count and grouping?
  • Scale as an experimental axis — existing benchmarks use ≤100 workers and don't vary n. Run all defences at n = 25, 50, 100, 200, 500 with fixed batch size. Plot accuracy vs n.
  • Two classes, two scaling behaviours — show that averaging-based methods (FedAvg, Trim-Mean, Median) improve or hold as n grows, while selection-based methods (Krum, Multi-Krum, Bulyan, SignGuard) degrade. This is a systematic finding about a class, not a single algorithm.
  • Isolate noise from heterogeneity — Karimireddy (ICLR 2022) frames bucketing as fixing non-IID. Run the same experiments with IID data and show the noise failure mode exists independently. Bucketing fixes both, but they are distinct problems.
  • Grouping as the equaliser — show that after grouping, both classes converge to similar performance. Grouping removes the scaling disadvantage of selection-based methods.
  • Practical guideline — for n workers with batch size b, what group size k is needed? The Phase 6 sensitivity grid (k × nbyz) can answer this empirically.
  • Grouping as a constraint, not a choice — all existing bucketing work (Karimireddy, Zhu) introduces grouping voluntarily for algorithmic benefit. In secure aggregation settings, grouping is imposed by the privacy protocol — the server never sees individual gradients. This changes the research question from "should you bucket?" to "given that you must group for privacy, what are the consequences for Byzantine robustness?" No existing paper starts from this constraint.
Requires but not yet available: Multi-Krum, Bulyan, Trim-Mean, Median implementations in the codebase. Currently have: FedAvg, Krum, ShieldFL, SignGuard.
Risk: if the finding only holds for Krum and not the broader class, the contribution narrows to a replication of Karimireddy with larger n. The scale sweep across multiple methods is what makes or breaks this.
