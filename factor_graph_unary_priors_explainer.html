Factor graph: current design vs. unary priors from latent_variables

Simplified view with 6 workers and 2 groups of 3 (your actual run has 500 workers × groups of 10). Circles are variables (one per worker, binary: benign / malicious). Squares are factors.

1. Current graph (one round)what exists today

Only k-ary group factors exist — one per group, each connecting its 3 (in reality: 10) worker variables. No prior is attached to individual variables, so BP starts from uniform messages every round.

variable (1 per worker, 2 states) k-ary group factor (observation over k workers)
Problem with memory: across rounds, new group factors (new random shuffles) are appended to this same graph — it grows by ~50 factors per round and never shrinks. By round 26 it's carrying ~1,300 factors.
Problem with logic: the posterior latent_variables[i] = p_i produced at round t is stored, used for weighting, and then ignored by BP at round t+1. Each round's BP restarts from uniform beliefs.

2. With unary priors from last round's latent_variablesproposed

Same k-ary group factors, plus a tiny unary factor on each variable that encodes its prior probability of being malicious, seeded from last round's posterior. Each unary factor holds just 2 numbers: log(1-pi) and log(pi).

variable k-ary group factor unary prior factor (πi)
Each πi is a single-variable factor holding 2 log-potentials: [log(1 − pi), log(pi)]. It nudges vi's belief toward last round's posterior before BP even starts combining this round's observations. The graph now has 6 + 2 = 8 factors instead of 2, but next round it's still 8 (not 16) — memory stays bounded because we rebuild cleanly.

3. Temporal cycle: posterior becomes next round's priorBayesian recursion

This is the standard streaming-Bayesian pattern (same shape as a Kalman filter): the previous round's posterior summarizes history into a single number per worker, and that number becomes the next round's prior via unary factors.

Round-over-round: graph gets rebuilt from scratch every round (bounded memory), but the information from all previous rounds flows forward, compressed into each worker's scalar prior pi. History lives in the numbers, not in the graph structure.
